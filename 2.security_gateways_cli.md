# Hydragent Security Vault, Gateway Adapters, & CLI Integration

This specification outlines the detailed use cases, security mechanisms, and architecture for the Vault, Gateways, and CLI subsystems.

---

## 🔒 1. The Cryptographic Vault (Phase 6.4 Security)

The Vault is our secure credentials store, protecting sensitive API keys and tokens from external exploits (like prompt injection) and internal vulnerabilities (LLM visual access).

### Model-Blind API Keys (Injection Protection):
The LLM must *never* have access to its own API keys or external platform tokens. Keys are injected directly into the network client parameters during socket connection setup. The LLM's system prompt and workspace context do not contain these variables, preventing key leakage via prompt injection.

### Dynamic Key Expiry & Overwriting:
If a key expires or is compromised, the user can run:
```bash
hydragent vault set <scope> <credential>
```
This encrypts and overwrites the stale credential. The system will hot-reload the key in memory without requiring a kernel restart.

## 🌐 2. Secure Remote Vault Administration

To ensure key changes cannot be observed by the LLM (which reads Page/Draft Paper files), and to make it easy for remote users who cannot handle long cryptographic key files, we establish the following protocol:

```
  [Remote Administrator]                              [Hydragent Kernel]
           │                                                  │
           │───────────── Request Vault Write ───────────────►│ 
           │             (Sends scoped payload)               │ (Bypasses the LLM completely;
           │                                                  │  does not write to Draft Paper)
           │◄────────── Cryptographic Challenge ──────────────│
           │           (Random nonce + Timestamp)             │
           │                                                  │
           │────────── Signed Challenge Response ─────────────►│
           │           (Signed via User's Passphrase PIN)     │ (Verifies against PBKDF2/SHA256
           │                                                  │  hash stored in local config)
           │◄────────── Authorized Vault Write ───────────────│
```

### Complete Isolation & Passphrase Protocol:
1. **Out-of-Band LLM Bypass**: Vault operations never touch the event bus or conversational Draft Paper logs. The remote RPC command channel bypasses the Librarian (LLM) completely, executing directly at the kernel system layer. No conversation memory is created.
2. **User-Friendly Key Generation (Passphrase PIN)**: To eliminate the need for remote users to carry or manage heavy cryptographic private key files on mobile/remote devices:
   - During vault setup, the user inputs a memorable **Passphrase PIN**.
   - The kernel generates a derived key locally using **PBKDF2 with SHA256** and a unique salt, storing the hash in `config/security/admin_auth.hash`.
   - The remote administrator only needs to type their passphrase to sign the challenge locally. The adapter derives the signature from the passphrase, keeping authentication fast and keyboard-friendly.
3. **Dual-Slot Authentication & Recovery PIN Reset**: The vault metadata supports two parallel decryption slots (similar to LUKS disk encryption):
   - **Slot 0 (Passphrase PIN)**: Unlocks the master key using the user-memorable PIN.
   - **Slot 1 (Admin Key File)**: Unlocks the master key using a local physical cryptographic private key file (e.g. Ed25519).
   - **PIN Recovery/Reset**: If a user forgets their Passphrase PIN, they can authenticate locally or remotely by providing the Admin Key File, which grants access to overwrite and reset the Passphrase PIN slot.
4. **No Visual Key Echoes**: Remote reads of keys are blocked. The remote command channel supports `set`, `list` (names of scopes only), and `delete`, but never exposes a `get` command to output key contents.

## 🛰️ 3. The Multi-Interface Gateway (Adapters)

The Gateway acts as the bridge connecting external communication platforms (Telegram, Discord, Slack, etc.) to the Hydragent kernel.

```
  ┌───────────────────────────────────────────────────────────┐
  │         External Clients (Telegram, Discord, Slack)       │
  └─────────────────────────────┬─────────────────────────────┘
                                │ Channels: WebSockets / Webhooks
                                ▼
  ┌───────────────────────────────────────────────────────────┐
  │                 Channel Adapters (adapters/)              │
  │     - Parses messages into standardized JSON-RPC          │
  │     - Translates kernel responses back to client layout   │
  └─────────────────────────────┬─────────────────────────────┘
                                │ TCP Port 5000 (JSON-RPC)
                                ▼
  ┌───────────────────────────────────────────────────────────┐
  │             Gateway Router (hydragent-gateway)             │
  │     - Registers active adapters                           │
  │     - Performs Deduplication and Rate Limiting            │
  └───────────────────────────────────────────────────────────┘
```

### Core Mechanisms & Use Cases:
1. **Deduplication (`Deduplicator`)**: To prevent the same event from triggering multiple processing threads (e.g. a double-click or network resend on Telegram), the gateway indexes inbound messages by `request_id`, `channel_id`, and `user_id` inside an in-memory window of `1000` entries. Identical queries are silently dropped.
2. **Rate Limiting (`RateLimiter`)**: To protect the system and backend LLMs from API flooding, each registered channel adapter receives a dedicated rate limiter. Exceeding thresholds drops the messages at the routing gate before hitting the ReAct loop.
3. **Outbound Push Messaging (`push`)**: The `GatewayRouter` supports routing push messages from the dream cycle, cron scheduler, or background feeds. A push addressed to `*` is automatically broadcast across all active adapters, while a targeted push routes directly to its specific `channel_id` (e.g., sending a digest directly to Telegram).
4. **Asynchronous Isolation**: Adapters operate as distinct processes communicating asynchronously. If an external API goes down (e.g., Discord outages) or its adapter crashes, the kernel unregisters the channel and continues running the Slack, Telegram, and CLI adapters without interruptions.

### JSON-RPC API Protocol Contracts (Client Integration API):

To develop adapters or client interfaces, connect to `127.0.0.1:5000` (TCP, line-delimited JSON-RPC) and use the following API specifications:

#### 1. Registration (`gateway.register`)
- **Purpose**: Registers the adapter as an active channel.
- **Direction**: Client $\rightarrow$ Server
- **Request**:
  ```json
  {"jsonrpc": "2.0", "method": "gateway.register", "params": {"channel_id": "telegram"}, "id": "reg-1"}
  ```
- **Response**:
  ```json
  {"jsonrpc": "2.0", "result": {"status": "ok"}, "error": null, "id": "reg-1"}
  ```

#### 2. Submitting Input (`intent.submit`)
- **Purpose**: Submits a user prompt turn.
- **Direction**: Client $\rightarrow$ Server
- **Request**:
  ```json
  {
    "jsonrpc": "2.0",
    "method": "intent.submit",
    "params": {
      "page_id": "session-uuid",
      "channel_id": "telegram",
      "user_id": "user-uuid",
      "content": "Tell me a joke",
      "attachments": [],
      "metadata": {},
      "timestamp": 1700000000000,
      "priority": "normal"
    },
    "id": "intent-1"
  }
  ```
- **Intermediate Stream Notifications (Server $\rightarrow$ Client)**:
  - **Token Chunk**: `{"jsonrpc": "2.0", "method": "response.token", "params": {"token": "Why did "}}`
  - **Status Message**: `{"jsonrpc": "2.0", "method": "response.status", "params": {"status": "\n`[Executing search...]`\n"}}`
  - **Permission Prompt**: If a tool requires confirmation:
    ```json
    {
      "jsonrpc": "2.0",
      "method": "response.permission_request",
      "params": {
        "request_id": "req-uuid",
        "page_id": "session-uuid",
        "tool_id": "file_write",
        "params_summary": "Write output to log.txt",
        "tier": "Prompt",
        "expires_at_ms": 1700000030000
      }
    }
    ```
  - **Stream End**: `{"jsonrpc": "2.0", "method": "response.complete", "params": {}}`
- **Final Response**:
  ```json
  {
    "jsonrpc": "2.0",
    "result": {
      "page_id": "session-uuid",
      "content": "Why did the chicken cross the road? To get to the other side!",
      "format": "markdown",
      "consent_requests": [],
      "tool_calls_executed": []
    },
    "error": null,
    "id": "intent-1"
  }
  ```

#### 3. Granting Permission (`permission.respond`)
- **Purpose**: Approves or denies a pending tool execution request.
- **Direction**: Client $\rightarrow$ Server
- **Request**:
  ```json
  {"jsonrpc": "2.0", "method": "permission.respond", "params": {"request_id": "req-uuid", "approved": true}, "id": "perm-1"}
  ```
- **Response**:
  ```json
  {"jsonrpc": "2.0", "result": {"status": "ok"}, "error": null, "id": "perm-1"}
  ```

## 💻 4. Command Line Interface (CLI) Complete Commands

To prevent architectural overlap, interactive user interaction (`hydragent chat`) runs under the JSON-RPC gateway adapter on port `5000` (incorporating all memory recall and security sanitizer gates). The core `hydragent` CLI is strictly reserved for administrative tasks.

### Complete Administrative Command Tree:

```
hydragent [flags] <command>
├── serve                   # Starts the gateway (bus, dream worker, tool registry)
├── ps                      # Lists active running processes on the host
├── stop [pid]              # Stops one or all running instances
├── status                  # Shows gateway state, brain model, and dream worker status
├── test-brain [prompt]     # Stream test prompt through the brain to check routing
├── onboard                 # Guided first-time setup for configuring environment variables
├── update                  # Checks GitHub Releases and performs in-place upgrades
├── uninstall               # Wipes binary, environment directories, and PATH settings
│
├── vault <action>          # Administrative actions for the credentials vault
│   ├── init                # Initialize data/vault/.hydravault
│   ├── set <scope> <val>   # Store an encrypted credential
│   ├── get <scope>         # Retrieve credential (disabled on remote ports)
│   ├── list                # List credential scopes (no values)
│   └── delete <scope>      # Delete credential
│
├── memory <action>         # Operations for semantic facts database
│   ├── list                # List first 20 memories
│   └── clear               # Wipe all stored memories
│
├── embed compare <t1> <t2> # Compare cosine similarity between two strings
│
├── audit <action>          # Operations for Phase 6.1 Merkle audit chain
│   ├── list                # List audit events with limit, offset, or reverse options
│   ├── head                # Print current head hash of the audit chain
│   └── verify [flags]      # Check integrity of the chain and/or signatures
│
└── security <action>       # Phase 6 security diagnostics
    ├── status              # Status check of sanitizer patterns and taint policy
    ├── taint-check         # Dry-run validation of taint checks
    ├── taint-policy        # Show currently active taint policies
    ├── sanitizer-scan <t>  # Scan text against prompt injection patterns
    ├── sanitizer-patterns  # List loaded sanitizer patterns
    ├── vault-status        # Summary of vault path and mlock status
    ├── vault-init          # Direct initializer for vault
    ├── vault-rotate-pass   # Rotates master vault password
    └── vault-rotate-col    # Rotates AES column-encryption key
```
