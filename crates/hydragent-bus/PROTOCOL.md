# Hydragent Event Bus Wire Protocol (v1.0.0)

This document describes the inter-process communication protocol used by Hydragent components to exchange messages between the Rust core and various channel adapters.

---

## 1. Transport Layer
- **Protocol**: TCP (Transmission Control Protocol) over IPv4 Loopback.
- **Host**: `127.0.0.1` (localhost).
- **Default Port**: `5000` (customizable via `BUS_PORT` in the `.env` file).
- **Framing**: Line-delimited JSON. Every single message (Request, Response, or Notification) is encoded as a single JSON object terminated by a newline character (`\n`).

---

## 2. Message Format (JSON-RPC 2.0)

All payloads adhere strictly to the JSON-RPC 2.0 specification.

### 2.1 Request Structure
Requests are sent from adapters (clients) to the Rust core (server) to invoke operations.

```json
{
  "jsonrpc": "2.0",
  "method": "method_name",
  "params": { ... },
  "id": "uuid-v4-string"
}
```

### 2.2 Response Structure
Responses are returned by the server upon finishing a request handler.

```json
{
  "jsonrpc": "2.0",
  "result": { ... },
  "error": null,
  "id": "uuid-v4-string"
}
```

### 2.3 Error Structure
If an operation fails, the response includes an `error` block:

```json
{
  "jsonrpc": "2.0",
  "result": null,
  "error": {
    "code": -32601,
    "message": "Method not found",
    "data": null
  },
  "id": "uuid-v4-string"
}
```

---

## 3. Supported Methods (Phase 1)

### 3.1 `intent.submit`
Sent by the adapter to submit a new user prompt or command turn.

- **Direction**: Adapter $\rightarrow$ Core
- **Params**: `IntentEvent` struct (serialized as JSON).
  ```json
  {
    "page_id": "session-uuid",
    "channel_id": "cli:default",
    "user_id": "local-user",
    "content": "What is the capital of France?",
    "attachments": [],
    "metadata": {},
    "timestamp": 1620000000000,
    "priority": "normal"
  }
  ```
- **Response Result**: `AgentResponse` struct.
  ```json
  {
    "page_id": "session-uuid",
    "content": "The capital of France is Paris.",
    "format": "markdown",
    "consent_requests": [],
    "tool_calls_executed": []
  }
  ```

---

## 4. Server-to-Client Stream Notifications

During long-running request handlers (like streaming token generation), the server streams intermediate notifications back through the active connection socket *before* returning the final `JsonRpcResponse` payload.

### 4.1 `response.token`
Emitted for each text token generated by the model in real time.

- **Direction**: Core $\rightarrow$ Adapter (Notification)
- **Params**:
  ```json
  {
    "jsonrpc": "2.0",
    "method": "response.token",
    "params": {
      "token": "Hello "
    }
  }
  ```

### 4.2 `response.complete`
Emitted to signal that token streaming has completed. The adapter should prepare to capture the final `JsonRpcResponse` containing the full message metadata.

- **Direction**: Core $\rightarrow$ Adapter (Notification)
- **Params**:
  ```json
  {
    "jsonrpc": "2.0",
    "method": "response.complete",
    "params": {}
  }
  ```
